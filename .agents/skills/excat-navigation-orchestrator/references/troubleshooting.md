# Troubleshooting — Navigation Orchestrator

| Issue | Cause | Action |
|-------|--------|--------|
| Sub-agent output rejected | JSON does not validate against schema | Re-run sub-agent; ensure all required fields and types match `references/*-schema.json`. Run `migration-work/navigation-validation/scripts/validate-output.js` to verify. |
| Gate blocks progress | Missing screenshot or uncertainty | Do not proceed. Request screenshot or clarification; do not infer. |
| Validation-agent recommends re-analysis | Mismatch in row/spacing/font/interaction/megamenu | Trigger re-analysis loop to relevant phase; do not silently adjust. |
| Script validate-output.js fails | Output file missing or invalid JSON | Ensure sub-agent returns only the prescribed JSON shape; no prose inside the payload. |
| How do we know critique validated? | Need proof that style loop ran | Check `style-register.json`: the 4 key components have `status: "validated"`, `allValidated: true`; each has critique proof (critiqueReportPath, screenshotSourcePath, screenshotMigratedPath, critiqueIterations ≥ 1). Rest may be `skipped`. |
| Style register has pending components | One or more of the 4 key components &lt; 95% similarity | Extract the exact styles from the source site so we match them precisely. Run Step 14 targeted critique (4 key components in parallel via key-component-agents); for any key component still pending, run nav-component-critique for that component; capture source + migrated screenshots, apply CSS/JS fixes, re-score until ≥ 95%; mark validated. Do not proceed until the 4 key components are validated (no key component may remain pending). |
| Schema register has pending items | Source vs migrated schema mismatch for some component | Fix implementation or re-extract migrated structure; re-run `compare-structural-schema.js --output-register=.../schema-register.json`. Do not proceed until schema-register allValidated. |
| Desktop renders as raw bullet lists | CSS not applied or wrong selectors | Extract the exact styles from the source site so we match them precisely. Implement full desktop styling: horizontal layout, list-style none, dropdown panels, CTA button style in blocks/header/header.css. Compare to source screenshot. |
| Megamenu has no images | hasImages: true but images not implemented | From source, identify megamenu image URLs; download or reference them; include in megamenu content/block so dropdown shows images, not text-only. |
| Structural similarity &lt; 95% | Migrated header structure differs from source | Extract the exact styles from the source site so we match them precisely. Re-extract from migrated page into `migrated-structural-summary.json`. Fix implementation to match source. Re-run `migration-work/navigation-validation/scripts/compare-structural-schema.js --threshold=95 --output-register=.../schema-register.json`. |
| Nav item assumed to have no hover | Link/redirect not tested for hover | Do not assume. Test hover and click separately for every nav item; set hasHoverBehavior and hasClickBehavior in phase-2/phase-3 from evidence. |
| nav.plain.html in wrong location | Wrong location | nav.plain.html must be at `content/nav.plain.html`, not workspace root. Move it. |
| nav.md found (not supported) | Legacy nav.md exists | nav.md is not supported. Create content/nav.plain.html instead. header.js fetches /nav.plain.html. Delete or rename nav.md. |
| Images missing from nav.plain.html | hasImages elements not downloaded | Before writing nav.plain.html, iterate every `hasImages: true` element from phase-2/3; download each image; include in nav.plain.html. If images are missing, go back and download. |
| Image file is empty (0 bytes) | Broken download — logo, flag, or thumbnail saved as 0-byte file | validate-nav-content.js BLOCKS when any referenced image has size 0. Re-download from source URL. Common causes: redirect to HTML error page, failed fetch, or placeholder. Check each file in content/images/ — especially logo and flags. |
| Image paths not found (absolute paths) | Used `/content/images/logo.svg` instead of `images/logo.svg` | path.resolve() treats leading `/` as filesystem root — validation looks at `/content/images/` on disk, not workspace. **Use relative paths only:** `<img src="images/filename.ext" alt="...">`. Works for both browser (resolves from page URL) and validation (resolves from nav file directory). External URLs (https://...) are fine. |
| Proceeded without all registers validated | Skipped validation gate | Read style-register.json and schema-register.json. If ANY item has status=pending or lastSimilarity below 95, STOP and fix. Run pre-confirmation gate checklist before asking customer. |
| Style register marks 70% as validated | Wrong: below 95% threshold | A component is validated ONLY at lastSimilarity >= 95. Revert status to pending; run critique again; apply fixes until 95%. |
| Registers list only high-level groups | Missing granular items | Both registers must list EVERY individual element from phase-1/2/3: every row, every element within each row, every megamenu column, every sub-menu. Re-build registers from phase files. |
| Hamburger → cross animation missing | Not tracked in phase-2/phase-4 | Extract the exact styles from the source site so we match them precisely. Re-analyze header: click hamburger icon, document animation type (`css-transform`, `svg-swap`, `class-toggle`), implement matching CSS transition in header.css. |
| Click/hover has no effect | Used JavaScript element.click() instead of Playwright | Use Playwright MCP's click and hover — NOT JavaScript. Many handlers only fire on real pointer events. Do NOT try JS click first; use Playwright from the start for hamburger, dropdowns, megamenu, nav items. |
| Unwanted overlay on megamenu | Implementation added overlay not in source | Extract the exact styles from the source site so we match them precisely. Check source site: does megamenu have backdrop/overlay? If NO, remove overlay from migrated header.css/header.js. If YES, match its opacity and color exactly. |
| Megamenu panel goes off viewport | Panel extends beyond screen bounds; measuredRight > viewportWidth or measuredLeft < 0 | Use getBoundingClientRect() on panel element — add measuredLeft, measuredRight, viewportWidth to panelLayoutDetails. Do NOT self-assess. Test at 1440, 1920, 1366, 1280, 1024px. Extract the exact styles from the source site so we match them precisely. Fix header.css (position, left, width, max-width) to match source positioning model. Re-run `compare-megamenu-behavior.js`. |
| Megamenu panel layout mismatch | viewportContained, overlayBehavior, CSS positioning, or alignment differs | Source and migrated must both have `panelLayoutDetails` per trigger with **measured values** (getBoundingClientRect) and **computed CSS** (getComputedStyle: position, left, width). Extract the exact styles from the source site so we match them precisely. Fix migrated panel CSS to use same positioning model as source. Re-run `compare-megamenu-behavior.js`. |
| Megamenu validated but has note describing mismatch | LLM shortcut: marked validated but added note (e.g. "not included in EDS", "source has X but migrated uses Y") | **Hook blocks:** Gate 20 and Stop check detect notes that describe source vs migrated differences. Do NOT mark items validated when adding such notes — fix the implementation to match source, then remove the note. |
| Header appearance mismatch | Header bar background/shadow on hover differs from source; or headerBackgroundBehavior.defaultState is transparent but header.css sets background-color on .nav-wrapper | Source `header-appearance-mapping.json` is created in Phase 2 (before implementation). If defaultState is transparent, ensure header.css does NOT set background-color on .nav-wrapper in default state — use transparent header template. Create `migrated-header-appearance-mapping.json` (same schema, including headerBackgroundBehavior). Run `compare-header-appearance.js`. Fix header.css to match source. |
| Mobile accordion not matching source | Wrong expand behavior | Extract the exact styles from the source site so we match them precisely. Re-analyze source mobile menu: is it `single-expand` (one section at a time) or `multi-expand`? Match the behavior in header.js. Check animation timing. |
| Mobile uses accordion but source uses slide-in | EDS default accordion used | Extract the exact styles from the source site so we match them precisely. Source uses slide-in-panel (menu slides left, sub-panel enters right with back button). Do NOT use accordion expand-in-place. Implement CSS translateX sliding panels. |
| Mobile menu items centered or with margins (source is full-width) | Desktop CSS (e.g. .default-content-wrapper justify-content: flex-end) applied in mobile without override | Check `phase-4-mobile.json` → `menuItemsWidthLayout`. If source is `full-width-flush`, migrated menu items must stretch edge-to-edge. In header.css add a mobile override: e.g. `@media (max-width: …) { .default-content-wrapper { justify-content: stretch; } .nav-menu a, .nav-menu li { width: 100%; margin-left: 0; margin-right: 0; } }` so items are full-width flush; remove or override any centering/margins that come from desktop rules. |
| Source has content not in nav.plain.html | LLM noted omission but proceeded | **Create `missing-content-register.json`** with each omission: `{ "items": [{ "location": "Concessionárias dropdown", "description": "Dealer search form", "resolved": false }] }`. Add the content to nav.plain.html, extract the exact styles from the source site so we match them precisely, set `resolved: true`, re-run validate-nav-content.js. The hook BLOCKS until all items resolved. Do NOT proceed with a note; the register + hook enforce the fix. |
| Mobile has extra items not in nav.plain.html | Mobile-only content (CTAs, sections) missing | **Create `mobile/missing-content-register.json`** with each omission. Add to nav.plain.html in a mobile-only section. Extract the exact styles from the source site so we match them precisely. In header.css: hide on desktop (`display: none` default), show in `@media (max-width: ...)` for mobile. Set `resolved: true`. Hook blocks until all resolved. Desktop view must NOT be impacted. |
| Megamenu text/links hardcoded in JS | Content belongs in nav.plain.html | Move ALL text content, link labels, category names, sub-menu items, and promotional copy from header.js into `content/nav.plain.html`. header.js should only parse the nav DOM and build the visual presentation — never generate text. |
| Locale: country names or flag URLs in header.js | Locale content belongs in nav.plain.html | Put ALL country names and flag image references in `content/nav.plain.html` (e.g. locale section with `<img src="images/flag-xx.svg" alt="Country"> Country` per entry). header.js must only READ from nav DOM and implement behavior (open/close, selection). header.css handles layout (e.g. multi-column grid). Run `validate-nav-content.js` after nav.plain.html changes; fix until it passes. |
| Images in custom/pipe-delimited format | Vehicle cards, megamenu thumbnails, or flags stored as pipe-delimited data or in header.js | **Use `<img src="images/filename.ext" alt="...">` in nav.plain.html.** No pipe-delimited format or hardcoded paths in header.js. header.js must READ image paths from the nav DOM. Run `validate-nav-content.js`; it BLOCKS until images use supported format. |
| migrated-row-elements-mapping created from intent, not from testing | Step 5a skipped or inferred | You MUST physically hover and click each row element (logo, nav, CTA, search, locale, hamburger, icons) on the **live migrated page** (localhost) and record what actually happens. Do NOT infer from nav.plain.html or header.js. Open the migrated page, test each element, then write migrated-row-elements-mapping.json. Run `compare-row-elements-behavior.js`. |
| Parallel agents for nav.plain.html + header.js + header.css | Causes mistakes in both | Do NOT use parallel agents. Work sequentially: nav.plain.html → validate-nav-content.js (exit 0) → header.js → header.css. |
